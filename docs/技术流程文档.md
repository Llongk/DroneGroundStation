# DroneGroundStation 技术执行流程文档

本文从程序启动、手机采集、STM32 遥测、飞控命令、数据库、QML 刷新到退出，说明数据在函数之间如何流动。字段和时序均以当前源码为准。

## 一、程序启动流程

```text
操作系统启动 EXE
  → main()
  → 设置 Qt Quick Controls Basic 样式
  → 创建 QGuiApplication
  → 创建 Backend
  → 创建 SensorBackend
  → 创建 HistoryDatabase
  → 建立后端间信号连接
  → 注册 QML 上下文属性
  → 加载 DroneGroundStation/Main.qml
  → 进入 Qt 事件循环
```

### 1. `Backend` 初始化

1. 创建 `QTcpServer`。
2. 连接 `newConnection` 信号。
3. 在 IPv4 任意地址的 8080 端口监听。
4. 调用 `preferredHotspotAddress()` 选择手机可访问地址。
5. 生成 `http://<IPv4>:8080/`。
6. 创建单次触发的 8 秒手机在线计时器。

### 2. `SensorBackend` 初始化

1. 创建 `QTcpSocket`。
2. 连接 `connected`、`disconnected`、`readyRead` 和错误信号。
3. 创建约 2 秒重连计时器。
4. 创建 3.5 秒连接超时计时器。
5. 创建 3.5 秒遥测看门狗。
6. 通过事件循环异步开始首次连接，避免构造函数阻塞 QML 启动。

### 3. `HistoryDatabase` 初始化

1. 确定便携数据库目录。
2. 创建 `database` 文件夹。
3. 必要时迁移旧应用数据目录中的数据库。
4. 使用唯一连接名打开 SQLite。
5. 设置 `journal_mode=WAL`、`synchronous=NORMAL`、`foreign_keys=ON`。
6. 创建 `phone_data`、`stm32_data` 和索引。
7. 订阅 `gpsChanged`、`sensorChanged`、`telemetryChanged`。

### 4. QML 对象注册

```text
backend          → 手机 GPS、姿态、HTTP 状态
sensorBackend    → 手机温湿度、STM32 遥测和命令
historyDatabase  → 会话、历史查询和异常查询
```

三个对象明确设置为 `CppOwnership`。页面栈销毁某一页面时，只销毁页面本身，不能销毁后端。

## 二、登录与会话流程

```text
用户输入账号密码
  → LoginPage 校验演示账号
  → historyDatabase.startSession(username)
  → 生成 session_id
  → 保存 username
  → emit sessionChanged()
  → Main.qml push(MainPage)
```

会话标识由当前时间和随机 UUID 后缀组成，用于把一次登录期间产生的记录组织在一起。手机和 STM32 表共享此标识，但不会共享数据行。

## 三、手机网页获取流程

### 1. TCP 接入

```text
浏览器连接 Qt:8080
  → QTcpServer::newConnection
  → Backend::newConnection()
  → 取出 pending QTcpSocket
  → 为 socket 建立请求缓存
  → 连接 readyRead/disconnected
```

HTTP 本质上运行在 TCP 上。一条请求头和请求体可能分多次到达，因此后端在数据完整前不能提前解析 JSON。

### 2. 路由处理

`Backend::readData()` 执行：

1. 读取当前 socket 可用字节并追加缓存。
2. 查找 `\r\n\r\n` 确定 HTTP 头结束位置。
3. 解析请求行、方法、路径和 `Content-Length`。
4. 等待完整请求体到达。
5. 根据方法和路径路由。
6. 调用 `sendResponse()` 返回完整 HTTP 响应。
7. 请求结束后断开短连接。

### 3. 页面请求

```text
GET / 或 GET /index.html
  → 从 Qt Resource 读取 index.html
  → Content-Type: text/html
  → markPhoneActivity()
  → 返回页面
```

网页被编译进 EXE，因此运行时不依赖外部 `index.html` 路径。

## 四、手机 GPS 上传流程

```text
浏览器 Geolocation/传感器 API
  → 组装 GPS JSON
  → fetch POST /gps
  → Backend::readData()
  → Backend::processGpsPayload()
```

`processGpsPayload()` 的处理顺序：

1. 将请求体解析为 JSON object。
2. 读取经纬度以及可选飞行属性。
3. 拒绝非有限数值。
4. 拒绝越界纬度、经度和 `(0, 0)`。
5. 检查定位精度是否适合实时使用。
6. 使用球面距离计算与上个有效点的距离。
7. 根据定位精度生成约 3～12 m 静止死区。
8. 小于死区时保持旧位置，抑制静止漂移。
9. 对大距离跳点计算隐含速度。
10. 同时满足距离和极端速度条件时拒绝跳点。
11. 更新经纬度、海拔、电量、姿态、精度、时间和速度。
12. 调用 `markPhoneActivity()`。
13. 发出 `gpsChanged()`。

### 信号下游

```text
gpsChanged()
  ├─ QML 文本绑定自动刷新
  ├─ MapView 追加有效位置并更新飞机标记
  ├─ FlightCurve 追加实时样本
  ├─ MainPage::updateWarnings()
  └─ HistoryDatabase::recordPhoneGps()
      → insertPhoneSnapshot("gps")
      → INSERT phone_data
```

## 五、手机温湿度上传流程

```text
网页获取温湿度
  → POST /sensor
  → Backend::processSensorPayload()
  → 校验 temperature/humidity
  → emit sensorDataReceived(temp, hum)
  → SensorBackend::updateSensorData()
  → emit sensorChanged()
```

`sensorChanged()` 同时触发主页卡片、告警计算和 `recordPhoneSensor()`。手机温湿度只存入手机状态，不会覆盖 STM32 的 DHT/SHT 字段。

## 六、手机在线租约与设备互斥

HTTP 是短连接，页面每次 `fetch` 都可能建立新 socket。程序用“最近有效活动时间”表示手机在线：

```text
有效手机请求
  → markPhoneActivity()
  → phoneConnected = true
  → 8 秒计时器重新开始
  → phoneConnectionChanged()
  → SensorBackend::setPhoneDeviceActive(true)
```

进入手机活动状态时：

- 停止 STM32 重连计时器。
- 停止连接超时计时器。
- 停止遥测看门狗。
- 终止现有 STM32 TCP 连接。
- 清理待确认命令状态。

8 秒没有新活动时：

```text
租约超时
  → phoneConnected = false
  → phoneConnectionChanged()
  → setPhoneDeviceActive(false)
  → scheduleReconnect()
```

## 七、STM32 AP 地址发现流程

`discoverApServerAddress()` 遍历网络接口和地址项：

1. 排除未启用、回环和无效接口。
2. 只处理 IPv4。
3. 读取地址与子网掩码。
4. 计算网络地址。
5. 取子网第一个主机地址作为常见 AP 网关。
6. 优先选择 `192.168.4.x` WLAN 候选。
7. 没有可用候选时保持“查找热点”状态并稍后重试。

自动发现解决了不同硬件或热点网段下 IP 不一致的问题。页面仍保留手动目标接口用于调试，但正常使用不需要写死 IP。

## 八、STM32 TCP 建连与重连流程

```text
attemptConnection()
  ├─ 若手机活动：直接返回
  ├─ 自动模式：discoverApServerAddress()
  ├─ 校验目标 IPv4 和端口
  ├─ socket.abort() 清理旧连接
  ├─ connectToHost(host, 8080)
  └─ 启动 3.5 秒连接超时
```

连接成功：

```text
QTcpSocket::connected
  → socketConnected()
  → 停止连接超时和重连计时器
  → 保存 peerAddress
  → 更新状态
  → 启动遥测看门狗
  → emit telemetryConnectionChanged()
```

连接断开或错误：

```text
socketDisconnected()/socketError()
  → 停止看门狗
  → 清空接收缓存
  → 清除有效遥测/待确认命令状态
  → 更新界面状态
  → scheduleReconnect()
  → 约 2 秒后 attemptConnection()
```

重连全部由计时器触发，不允许在主线程里循环连接或 `waitForConnected()`，否则 QML 会无响应。

## 九、TCP 粘包与拆包流程

TCP 只保证字节顺序，不保留发送端的消息边界。可能出现：

- 半个 JSON 到达。
- 一次到达多个 JSON。
- 一个完整 JSON 加下一帧的一部分。

处理算法：

```text
readyRead
  → m_buffer += socket.readAll()
  → 若缓存超过 64 KiB：清理并报错
  → while 找到 '\n'
       取出一行
       删除末尾 '\r'
       忽略空行
       processTelemetryFrame(frame)
  → 保留最后一个不完整片段等待下次 readyRead
```

发送端必须统一以 `\r\n` 结束每帧。仅发送 JSON 而没有结束符，会使接收端一直等待帧边界。

## 十、STM32 遥测解析流程

`processTelemetryFrame()` 先解析 JSON，再按 `type` 区分命令确认与遥测。

遥测字段检查：

```text
id, proto, seq,
lat, lng, heading, target_alt, speed,
rth_lat, rth_lng, rth_distance,
dht_temp, dht_hum, sht_temp, sht_hum,
mcu_temp, mode, alarm, battery, ts
```

处理步骤：

1. JSON 必须是 object。
2. 所有数值字段必须存在且为有限 number。
3. 经纬度、航向、电量等执行合理范围检查。
4. 航向归一化到一圈范围。
5. 更新全部 STM32 属性。
6. 标记 `telemetryValid=true`。
7. 重启 3.5 秒遥测看门狗。
8. 更新本机接收时间。
9. 发出 `telemetryChanged()`。

### 遥测下游

```text
telemetryChanged()
  ├─ Stm32Page 参数卡片刷新
  ├─ Stm32Page::updateWarnings()
  ├─ Stm32MapView 追加位置轨迹
  ├─ Stm32FlightCurve 追加曲线样本
  └─ HistoryDatabase::recordStm32Telemetry()
      → INSERT stm32_data
```

地图是否移动取决于连续遥测的 `lat/lng` 是否变化。按钮 ACK 成功但固件仍发送相同坐标时，轨迹不会凭空移动。

## 十一、飞控命令下发流程

### 1. QML 触发

底部按钮调用：

```text
sendFlightCommand(command, headingInput.value,
                  altitudeInput.value, speedInput.value)
```

### 2. C++ 前置校验

1. 手机当前不能占用设备通道。
2. TCP socket 必须为 ConnectedState。
3. 当前不能已有 `commandPending`。
4. 命令必须属于允许集合。
5. 航向范围为 `[0, 360)`。
6. 高度范围为 `[0, 500]`。
7. 速度范围为 `[0, 50]`。

任一校验失败时返回 `false`，更新 `commandStatus`，但不应阻塞或结束应用。

### 3. 命令编码

```text
command_seq = 单调递增命令序号
type = flight_command
proto = 1
id = 当前设备 ID 或默认 UAV_01
control = free_flight ? autonomous : manual
use_preset_route = free_flight
heading/target_alt/speed = 页面输入
ts = 当前毫秒时间戳
```

序列化后追加 `\r\n`，调用异步 `QTcpSocket::write()`。程序不会使用阻塞式 `waitForBytesWritten()`。

## 十二、命令 ACK 处理流程

```text
STM32 收到 flight_command
  → 校验并执行/拒绝
  → 返回 command_ack + CRLF
  → Qt readTelemetryData()
  → processTelemetryFrame()
```

ACK 必须满足：

- `type` 为 `command_ack` 或 `ack`。
- `command_seq` 存在。
- `command_seq` 等于当前待确认序号。
- `ok` 为布尔值。
- `message` 用于向用户解释结果。

匹配成功后清除待确认状态。`ok=true` 时，`free_flight` 切为自主模式，其他命令切为人工模式；`ok=false` 保持原控制模式并显示拒绝原因。

3 秒未收到匹配 ACK 时解除等待并显示超时。超时不等于固件一定没有执行，因此协议实现应确保每条命令都返回 ACK。

## 十三、实时地图与曲线更新

### 手机地图

```text
Backend::gpsChanged
  → QML Connections/onGpsChanged
  → 检查有效经纬度
  → 更新飞机坐标
  → 按策略追加轨迹
  → 跟随模式下调整地图中心
```

### STM32 地图

```text
SensorBackend::telemetryChanged
  → Stm32MapView 收到新坐标
  → 更新飞行器图标和航向
  → 追加实时航线
  → 跟随模式下更新中心
```

用户拖动和缩放地图时可以暂时脱离自动跟随；重新启用跟随后以最新有效坐标居中。OSM 底图和轨迹图层相互独立，瓦片下载失败不会禁止轨迹数据更新。

## 十四、SQLite 写入流程

### 手机记录

```text
gpsChanged/sensorChanged
  → 检查数据库已打开且 session_id 非空
  → 读取 Backend + SensorBackend 当前手机状态
  → 绑定 prepared query 参数
  → INSERT phone_data
  → emit historyChanged()
```

`event_type` 用来区分 GPS 更新和 sensor 更新。每行仍保存当时完整快照，便于历史页面统一绘图。

### STM32 记录

```text
telemetryChanged
  → 检查 telemetryValid 和会话
  → 绑定整帧遥测字段
  → INSERT stm32_data
  → emit historyChanged()
```

只有通过协议校验的帧才写数据库，坏 JSON 不会污染历史记录。

## 十五、历史记录查询流程

### 1. 会话摘要

```text
HistoryPage 打开
  → historyDatabase.getHistoryRecords()
  → 分别聚合 phone_data 和 stm32_data
  → 返回 source/session/time/count/start/end/battery/alarmCount
  → HistoryTable 显示
```

### 2. 会话详情

```text
用户选择一条记录
  → HistoryPage::selectRecord(sessionId, source)
  → getSessionData(sessionId, source)
  → 按 recorded_ms 顺序返回 QVariantList
  → 拆分经纬度、高度、温度、湿度数组
  → 刷新轨迹和图表
```

### 3. 异常点

```text
getAbnormalData(sessionId, source)
  → 查询会话全部样本
  → 按来源应用阈值
  → 合并同一样本的异常原因
  → 返回时间、坐标、数值和原因
  → StatusPanel/告警视图显示
```

异常判断是只读分析，不修改数据表。

## 十六、数据库迁移与 WAL

SQLite WAL 模式可能同时存在：

```text
FlightHistory.db
FlightHistory.db-wal
FlightHistory.db-shm
```

安全迁移方法：

1. 完全退出地面站。
2. 确认进程已结束。
3. 复制整个 `database` 文件夹，或至少一起复制上述现存文件。
4. 在新设备把文件放到源码项目根或 EXE 同级目录。
5. 启动后检查 `databasePath`。

若只迁移源码、不迁移数据库，首次运行会创建一个新的空数据库，这属于正常行为。

## 十七、QML 页面生命周期

`Main.qml` 的 `StackView` 负责页面切换：

```text
LoginPage
  └─ push MainPage
       ├─ push HistoryPage ─ pop 返回
       └─ push Stm32Page  ─ pop 返回
```

页面可被销毁重建，但 `backend`、`sensorBackend` 和 `historyDatabase` 始终由 C++ 主函数持有。QML `Connections` 只订阅信号，不改变对象所有权。

高频信号处理要求：

- 不执行同步网络操作。
- 不在回调中死循环。
- 曲线和轨迹应限制长期样本数量。
- 页面离开后不保留指向已销毁 QML 对象的 C++ 指针。

## 十八、异常与恢复机制

| 异常 | 检测位置 | 恢复方式 |
|---|---|---|
| HTTP 请求不完整 | `readData()` | 等待剩余字节 |
| HTTP/JSON 非法 | `process*Payload()` | 返回 400，不更新状态 |
| GPS 静止漂移 | `processGpsPayload()` | 死区保持旧位置 |
| GPS 极端跳点 | `processGpsPayload()` | 丢弃该点 |
| STM32 地址未发现 | `attemptConnection()` | 延时重新扫描 |
| TCP 建连无响应 | 连接超时计时器 | abort 后重连 |
| 已连接无遥测 | 遥测看门狗 | 断开后重连 |
| TCP 数据无边界 | 接收缓存 | 等待 `\n`，超上限清理 |
| 遥测字段非法 | `processTelemetryFrame()` | 丢弃帧，保留旧状态 |
| ACK 序号错误 | ACK 分支 | 忽略旧确认 |
| ACK 超时 | 命令超时计时 | 清除 pending 并提示 |
| OSM 无网络 | Qt Location | 轨迹继续更新，底图可能空白 |
| QML 加载失败 | `main.cpp` | 写启动诊断日志并退出 |

实时阈值异常由页面单独维护 `hasActiveAlarm` 状态，并用非阻塞 `SequentialAnimation` 驱动红色边缘呼吸闪动。动画不会改变网络处理节奏，也不会拦截地图或按钮输入；等待连接等状态提示不会启动动画。

## 十九、完整数据流总览

```text
手机浏览器
  ├─ GET index.html ───────────────────────────────┐
  ├─ POST /gps ─→ Backend ─→ gpsChanged ──────────┤
  └─ POST /sensor → Backend → SensorBackend ──────┤
                                                    ├→ MainPage/Map/Curve
                                                    └→ phone_data

STM32 AP/TCP Server
  ├─ telemetry JSON + CRLF
  │      → QTcpSocket → buffer → frame parser
  │      → SensorBackend properties
  │      ├→ Stm32Page/Map/Curve
  │      └→ stm32_data
  │
  └─ command_ack + CRLF
         → sequence match → command state → QML

QML 飞控按钮
  → parameter validation
  → flight_command JSON + CRLF
  → QTcpSocket::write
  → STM32 executes and continues telemetry

历史页面
  → session summary
  → source/session detail
  → charts/path/abnormal points
```

## 二十、调试检查顺序

### 手机链路

1. `serverRunning` 是否为 true。
2. `accessUrl` 是否属于当前网段。
3. `/health` 是否可访问。
4. 手机权限是否允许。
5. POST 是否返回 `{"ok":true}`。
6. `gpsChanged/sensorChanged` 是否持续发生。
7. `currentSessionId` 是否非空。
8. `phone_data` 是否增长。

### STM32 链路

1. 电脑是否真的连接 STM32 AP。
2. 自动目标是否为 AP 网关和 8080。
3. TCP 是否 connected。
4. STM32 是否持续以 `\r\n` 结束遥测。
5. `sequence` 是否递增。
6. 坐标是否实际变化。
7. 按钮后 STM32 是否收到 `flight_command`。
8. ACK 是否带相同 `command_seq`。
9. ACK 后是否继续发送遥测。
10. `stm32_data` 是否增长。

### 崩溃或无响应

1. 使用 Debugger 获取调用栈，而不只观察最后一条日志。
2. 确认没有 `waitForConnected`、`waitForReadyRead` 或无限循环进入 UI 线程。
3. 检查 QML 启动诊断日志。
4. 检查 StackView 页面是否存在冲突 anchors。
5. 检查 C++ 上下文对象仍为 `CppOwnership`。
6. 检查接收缓存和曲线数组是否无限增长。

## 二十一、扩展协议时的约束

增加遥测字段时应同步修改：

1. STM32 JSON 构建函数。
2. `SensorBackend` 属性、成员和 getter。
3. `processTelemetryFrame()` 字段校验。
4. QML 参数、曲线或告警展示。
5. `stm32_data` 表结构和 INSERT。
6. 历史查询映射。
7. 异常阈值（如需要）。
8. README 和协议文档。

增加飞控命令时应同步修改：

1. QML 按钮和参数。
2. Qt 允许命令集合。
3. STM32 命令解析枚举。
4. 模式切换逻辑。
5. ACK 返回与错误信息。
6. 联调测试用例和文档。

保持这些层同步，才能避免“界面已有按钮但固件认为无效命令”或“实时曲线正常但历史字段缺失”等跨层问题。
